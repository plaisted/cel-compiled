## Phase 2b: Fixups

Corrections to completed Phase 1/2 work that must land before Phase 3 builds on top of these foundations.

- [x] 2b.1 [delegate] Fix lossy numeric comparison in `CelRuntimeHelpers` ‚Äî `NumericEquals(long, double)`, `NumericEquals(ulong, double)`, `NumericCompare(long, double)`, and `NumericCompare(ulong, double)` currently cast the integer side to `double` and compare. This is incorrect for integer values above 2^53 where `double` loses precision (e.g., `(1L << 53) + 1` rounds when cast to `double`, making two distinct integers compare equal). Rework these methods to use exactness checks: verify the `double` is an exact integer (no fractional part, within representable range) before comparing, and use integer-space comparison when the `double` is integral. Add regression tests for values at `2^53 ¬± 1`, `ulong` values above `2^53`, and fractional doubles near integer boundaries (e.g., `9007199254740993L == 9007199254740993.0` should be `true`, `9007199254740993L == 9007199254740992.0` should be `false`).
- [x] 2b.2 [delegate] Harden `CelEquals`/`CelCompare` stubs ‚Äî `CelEquals(object?, object?)` currently falls back to `object.Equals` (CLR semantics, not CEL semantics) and `CelCompare(object?, object?)` throws `NotImplementedException`. Both are unsafe integration points that could silently produce wrong results or crash before tasks 3.2/3.3 fully implement them. Change both stubs to throw `CelRuntimeException` with a clear message indicating they are not yet implemented, so incorrect usage is caught immediately rather than silently returning wrong answers. This is a temporary safety net until 3.2 and 3.3 replace the implementations.
- [x] 2b.3 [delegate] Fix parser error message interpolation ‚Äî three error messages in `CelParser.cs` use escaped braces (`{{x}}`) instead of interpolation (`{x}`), causing literal brace text to appear instead of actual values: (1) `ReadOperatorOrPunctuator` line with `$"Unexpected character '{{c}}'"` should be `$"Unexpected character '{c}'"`, (2) `ParsePrimary` default case with `$"Unexpected token {{token.Type}}"` should be `$"Unexpected token {token.Type}"`, (3) `Expect` method with `$"Expected {{type}} but got {{token.Type}}"` should be `$"Expected {type} but got {token.Type}"`. Add at least one negative parser test that asserts the rendered error message contains the actual token or character text, not just that an exception was thrown.
- [x] 2b.4 [delegate] Fix bytes literal UTF-16 truncation ‚Äî `ReadBytes` and `ReadTripleQuotedBytes` in `CelParser.cs` use `(byte)_input[_pos]` for non-escaped characters, which truncates UTF-16 code units to a single byte. This silently corrupts any non-ASCII character. `ReadRawBytes` and `ReadTripleQuotedRawBytes` correctly use `Encoding.UTF8.GetBytes`. Fix `ReadBytes` and `ReadTripleQuotedBytes` to use `Encoding.UTF8.GetBytes` for plain characters (matching the raw variants). Add tests for non-ASCII bytes literals in both escaped and raw forms to verify consistent behavior.

## Phase 3: Compiler ‚Äî Strict Operator Semantics

Rework the compiler's operator lowering to follow CEL dispatch rules.

- [x] 3.0 [noplan] Introduce `CelResult<T>` non-throwing error channel ‚Äî define a lightweight value-type wrapper (struct) that carries either a success value or a CEL error (e.g., `no_matching_overload`, division by zero) without throwing exceptions. Sub-expressions that can produce CEL errors should compile into code that returns a `CelResult` instead of throwing. Design the type so it integrates cleanly with `Expression.Block`/`Expression.Variable` in compiled expression trees. This is the foundation for error-absorption in `&&`/`||` (tasks 3.4/3.5) and comprehension macros (Phase 7). Add unit tests for the result type itself. Keep exceptions for truly exceptional failures (e.g., stack overflow, invalid program state).
- [x] 3.1 Rework arithmetic operators (`+`, `-`, `*`, `/`, `%`) ‚Äî reject mixed numeric types with `no_matching_overload`. Same-type arithmetic only: `long+long`, `ulong+ulong`, `double+double`. String concatenation (`string+string`) and list concatenation (`list+list`) remain supported. Add overflow checking for `int` and `uint` arithmetic (CEL spec requires error on overflow). Update existing tests, add new tests for rejection: `1 + 1u` ‚Üí error, `1 + 1.0` ‚Üí error.
- [x] 3.2 Rework equality operators (`==`, `!=`) ‚Äî fully implement `CelRuntimeHelpers.CelEquals` (replacing the placeholder from 2b.2) as the **fallback** for dynamic/unknown-type operands. The compiler should prefer **typed specialization**: when both operand types are known at compile time and are the same primitive, emit `Expression.Equal` directly (no boxing). When operand types are known cross-type numerics (`long` vs `ulong`, `long` vs `double`, etc.), emit calls to the typed `NumericEquals` overloads (still no boxing). Fall back to `CelEquals(object?, object?)` only for genuinely dynamic cases (e.g., JSON-backed values). `null == null` is `true`. Cross-type non-numeric comparison (e.g., `string == int`) returns `false` (not error). Update and add tests.
- [x] 3.3 Rework ordering operators (`<`, `<=`, `>`, `>=`) ‚Äî fully implement `CelRuntimeHelpers.CelCompare` (replacing the placeholder from 2b.2) as the **fallback** for dynamic/unknown-type operands. The compiler should prefer typed specialization: same-type primitive ordering emits `Expression.LessThan`/etc. directly; known cross-type numerics emit typed `NumericCompare` overloads; `CelCompare(object?, object?)` is fallback only. String and bytes use lexicographic ordering. Cross-type non-numeric comparisons raise `no_matching_overload`. Update and add tests.
- [x] 3.4 [noplan] Implement `&&` with error-absorption semantics ‚Äî compile `a && b` using the `CelResult` channel from task 3.0. Both sides evaluate into `CelResult<bool>` locals (no throwing). Apply the CEL absorption rule: if either side's result is `false`, return `false` regardless of whether the other side errored. If both sides error, propagate one of the errors. Do NOT use `Expression.TryCatch`. Add tests: `false && (1/0 > 0)` ‚Üí `false`, `(1/0 > 0) && false` ‚Üí `false`, `true && (1/0 > 0)` ‚Üí error.
- [x] 3.5 [noplan] Implement `||` with error-absorption semantics ‚Äî same pattern as `&&` but using `CelResult` channel for `true` absorbing errors. Add tests: `true || (1/0 > 0)` ‚Üí `true`, `(1/0 > 0) || true` ‚Üí `true`, `false || (1/0 > 0)` ‚Üí error.
- [x] 3.6 [delegate] Verify ternary operator ‚Äî ternary already uses `Expression.Condition` which only evaluates the selected branch. Verify this with a test: `true ? 1 : 1/0` ‚Üí `1` (no error). `false ? 1/0 : 2` ‚Üí `2`.
- [x] 3.7 [delegate] Implement unary minus for `uint` ‚Äî `-uint_value` should produce a `no_matching_overload` error per the CEL spec (negation is not defined for `uint`). Add test.
